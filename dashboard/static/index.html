AFL++ Coverage Metrics Explained

Edges Found — The number of unique control-flow edges (source block → destination block pairs) discovered. This is the primary coverage metric in AFL++. More edges = deeper code exploration. ISOBMFF has roughly 10,000–20,000 reachable edges.

Bitmap Coverage % — AFL++ uses a 64KB (65,536-bucket) hash map to track edges. This percentage shows how much of that hash map is populated. This is NOT the same as source-line coverage. Because the hash map uses 65,536 buckets but real programs have far fewer edges, a value of 10–40% is completely normal and can represent very good coverage. Two different edges can hash to the same bucket (alias), so high bitmap% doesn't mean all code was reached.

Exec/s — How many test cases AFL++ runs per second. Higher is better. ISOBMFF writes to /dev/shm (RAM) to avoid disk I/O, so ~200–800 exec/s is typical. Lower values mean ISOBMFF is spending more time on complex inputs.

Corpus — The current set of inputs kept by AFL++ because they triggered new coverage. Starts from 368 seed files (minimized from 157,872 libFuzzer corpus via afl-cmin). Grows as new interesting inputs are discovered.

Cycles Done — How many times AFL++ has cycled through its entire corpus. Low cycle count (0–2) means AFL++ is still in the initial exploration phase. Higher cycles mean the fuzzer is in a "maintenance" mode, revisiting known inputs with mutations.

For source-line coverage (like libFuzzer's HTML reports), run make coverage inside the fuzzer container. This runs the AFL++ corpus through a libFuzzer-instrumented build and generates an HTML line-by-line report.
